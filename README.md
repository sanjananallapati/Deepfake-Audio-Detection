# Deepfake Audio Detection 🎙️🛡️

Classify a speech recording as **Genuine (Human)** or **Deepfake (AI-Generated)**.

This repository is a complete, reproducible solution for *Problem Statement 2 —
Deepfake Audio Detection*: a PyTorch deep-learning pipeline (data → log-mel
features → CNN → calibrated probability), a command-line tool to test new audio,
an automatically generated performance report, and a hosted **Streamlit** web app
that takes an audio file and returns the verdict plus a confidence score.

> **Label convention used everywhere:** `0 = Genuine (Human)`, `1 = Deepfake
> (AI-Generated)`. The model's *detection score* is the deepfake probability, and
> the **Equal Error Rate (EER)** is computed with *deepfake* as the positive class.

## Approach at a glance

| Stage | Choice | Why |
|---|---|---|
| Input | 16 kHz mono, fixed 4 s (pad/crop) | Uniform tensors; 4 s captures enough context cheaply |
| Feature | 128-bin **log-mel spectrogram** (instance-normalised) | Vocoder/TTS artefacts (over-smoothed spectra, unnatural harmonics/phase) show up clearly in the time–frequency plane |
| Augmentation | **SpecAugment** (freq/time masking), train-only | Cheap regulariser that improves generalisation to unseen synthesis methods |
| Model | Compact **VGG-style 2-D CNN** + global average pool | Learns local spectro-temporal artefacts; global pool makes it length-agnostic and removes flatten-dim fragility |
| Selection | Checkpoint at **lowest validation EER** | EER is the metric the brief is strictest on (≤ 12 %) |
| Output | `softmax` → P(deepfake); threshold 0.5 | Simple, calibrated, interpretable confidence |

Everything is **feature-based + end-to-end hybrid** in spirit: hand-chosen
front-end (log-mel) feeding a fully learned classifier.

## Pipeline

```
        ┌─────────────┐   ┌──────────────────────┐   ┌───────────────────┐
audio → │ Load + mono │ → │ Pad/crop to 4 s @16k │ → │ Log-mel (128×251) │ →
 file   │  (librosa)  │   │  fixed-length tensor │   │  instance-norm    │
        └─────────────┘   └──────────────────────┘   └───────────────────┘
              ┌──────────────────────────┐   ┌──────────────────────────────┐
          →   │ SpecAugment (train only) │ → │ SpecNetCNN (2-D conv blocks) │ →
              └──────────────────────────┘   └──────────────────────────────┘
                   ┌─────────────────────┐   ┌───────────────────────────────┐
               →   │ AdaptiveAvgPool → FC│ → │ softmax → P(genuine),P(fake)  │
                   └─────────────────────┘   └───────────────────────────────┘
                                                          │
                                       verdict + confidence (CLI / Streamlit)
```

## Quick start

```bash
# 1. Environment
python -m venv .venv && source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt
# For a CPU-only or specific-CUDA torch build, install torch first from
# https://pytorch.org/get-started/locally/ (see requirements.txt header).

# 2. Get the data (see "Dataset setup") and check the layout
python scripts/prepare_dataset.py --root data/for-norm

# 3. Train
python -m src.train

# 4. Evaluate (writes reports/performance_report.md + figures)
python -m src.evaluate

# 5. Test a new clip
python -m src.predict path/to/clip.wav

streamlit run app/streamlit_app.py
```

## Dataset setup

This project targets **The Fake-or-Real (FoR) dataset**
(`kaggle.com/datasets/mohammedabdeldayem/the-fake-or-real-dataset`). The audio
itself is **not** committed to the repo. Download it and arrange it so each split
contains class sub-folders. The scanner is tolerant of naming — `real`/`fake`,
`genuine`/`spoof`, `bonafide`/`synthetic`, etc. (case-insensitive) all work — and
will also descend one extra level if needed.

Expected default layout (matches `config.yaml`):

```
data/
└── for-norm/
    ├── training/    ├── real/ *.wav   └── fake/ *.wav
    ├── validation/  ├── real/ *.wav   └── fake/ *.wav
    └── testing/     ├── real/ *.wav   └── fake/ *.wav
```

> The brief says to use the **`for-norm`** (normalised) variant's training split.
> If your download uses different folder names or nesting, either rename to the
> above or point the paths at your layout via `config.yaml` or the
> `--train-dir/--val-dir/--test-dir` flags. Run `scripts/prepare_dataset.py` to
> confirm the counts before training.

**Optional cross-dataset test:** ASVspoof 2019 (`asvspoof.org/index2019.html`)
can be used for supplementary training or generalisation testing — point
`--test-dir` at a folder laid out the same way (its `bonafide`/`spoof` names are
recognised automatically).

## Training

```bash
python -m src.train                                   # defaults / config.yaml
python -m src.train --epochs 40 --batch-size 64 --lr 5e-4
python -m src.train --train-dir data/for-norm/training --val-dir data/for-norm/validation
```

- If no `--val-dir` exists, a stratified 15 % validation split is carved off the
  training data.
- Class imbalance is handled with inverse-frequency **class weights** (disable
  with `--no-class-weights`).
- The **best checkpoint by validation EER** is saved to `models/best_model.pt`.
  The checkpoint is **self-contained** — it stores the weights *and* the exact
  audio/feature/model settings *and* the label map — so evaluation and inference
  never depend on `config.yaml` matching.
- Learning curves are written to `reports/figures/training_history.png` and the
  raw history to `reports/history.json`.

## Evaluation & the performance report

```bash
python -m src.evaluate                                # uses paths.test_dir
python -m src.evaluate --test-dir data/for-norm/testing --model models/best_model.pt
```

This runs batched inference over the test set and **auto-generates**:

- `reports/performance_report.md` — accuracy, EER, F1, per-class accuracy, the
  confusion matrix, and an explicit **PASS/FAIL** against the §5 thresholds.
- `reports/metrics.json` — the same numbers, machine-readable.
- `reports/figures/confusion_matrix.png` and `reports/figures/roc_curve.png`
  (ROC annotated with the EER operating point).

## Streamlit web app

```bash
streamlit run app/streamlit_app.py
```

Upload an audio file → it plays the clip, runs the model, and shows **Genuine**
or **Deepfake** with a confidence score, the two class probabilities, a deepfake-
probability bar, and (expandable) the exact log-mel spectrogram the model saw.

## Preprocessing & feature extraction

1. **Load & resample** to 16 kHz, mixed to mono (`librosa`).
2. **Fix length** to 4 s (64 000 samples): shorter clips are loop-padded, longer
   clips centre-cropped — so every input is the same size.
3. **Log-mel spectrogram:** STFT with `n_fft=1024`, `hop_length=256`,
   `win_length=1024`; **128** mel bands over 20 Hz–8 kHz; power → dB
   (`ref=max`, `top_db=80`). Output shape **(128, 251)**.
4. **Instance normalisation:** subtract mean / divide by std per clip, so absolute
   loudness doesn't matter.
5. **SpecAugment** (training only): up to 2 frequency masks (≤16 bins) and 2 time
   masks (≤24 frames) zeroed-out — a strong, cheap regulariser.

All settings live in `config.yaml` / `src/config.py` and are baked into each
saved checkpoint.

## Model architecture

`SpecNetCNN` — a compact VGG-style 2-D CNN (defined in `src/model.py`):

- **4 convolutional blocks**, channels `32 → 64 → 128 → 128`. Each block:
  `Conv3×3 → BatchNorm → ReLU` ×2, then `MaxPool(2)`, then `Dropout2d`.
- **Global `AdaptiveAvgPool2d(1)`** head → the classifier input depends only on
  channel count, never on clip length (so inference accepts any duration and
  there is no flatten-dimension to mismatch).
- **Classifier:** `Linear(128 → 64) → ReLU → Dropout → Linear(64 → 2)` producing
  two logits; `softmax(logits)[:, 1]` is the deepfake probability.
- ~0.6 M parameters — trains comfortably on a single GPU and runs on CPU.

**Training recipe:** Adam (`lr=1e-3`, `weight_decay=1e-4`), cross-entropy with
optional class weights, `ReduceLROnPlateau` on val loss, early stopping on val
EER (patience 7), mixed precision when CUDA is available.

## Evaluation metrics & verification thresholds

Per the problem statement, a submission is **valid only if both primary
thresholds are met** on the evaluation set; the secondary metrics are also
reported.

| Type | Metric | Required |
|---|---|---|
| Primary (§5) | Overall Accuracy | **≥ 80 %** |
| Primary (§5) | Equal Error Rate (EER) | **≤ 12 %** |
| Secondary (§4) | F1 Score (macro) | ≥ 80 % |
| Secondary (§4) | Per-Class Accuracy (each class) | ≥ 75 % |
| Secondary (§4) | Confusion Matrix | reported |

> **EER** is the operating point where the false-acceptance rate equals the
> false-rejection rate (lower = a more balanced, reliable detector). Accuracy is
> additionally judged on a **private hidden test set**, which is why the pipeline
> emphasises generalisation (instance norm, SpecAugment, class weights,
> EER-based model selection).

## Reproducibility

- A global seed (`config.train.seed`, default 42) seeds Python, NumPy and Torch.
- The full config is embedded in every checkpoint.
- `tests/test_pipeline.py` is an end-to-end smoke test on synthetic audio:

  ```bash
  pytest -q            # or: python tests/test_pipeline.py
  ```

  The feature/scan/metric tests run with no dependencies beyond the core stack;
  the model/training/inference tests run when `torch` is installed.

## Deployment

**Streamlit Community Cloud (free):**

1. Push this repo to GitHub.
2. Make `models/best_model.pt` available — either commit it via **Git LFS**
   (`git lfs install && git lfs track "*.pt"`) or host the weights externally
   and download them on startup.
3. On [share.streamlit.io](https://share.streamlit.io), create an app pointing at
   `app/streamlit_app.py`; it installs `requirements.txt` automatically.

> The default `torch` wheel can be large for free tiers. If you hit size/build
> limits, pin the **CPU-only** torch build (see `requirements.txt` header) to cut
> the install down substantially.

## Notes, limitations & ethics

- A statistical detector can be wrong — especially on synthesis methods, codecs,
  or noise conditions unlike its training data. Treat the output as **one signal,
  not proof**, and prefer reporting calibrated probabilities over hard labels for
  high-stakes decisions.
- Performance depends heavily on training-data diversity; broadening the spoof
  sources (e.g. adding ASVspoof) is the most reliable way to improve real-world
  robustness.
- This tool is intended for defensive/authenticity-verification use.
